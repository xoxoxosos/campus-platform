## 一、需求概述

### 1.1 功能需求
- **多端分离登录**：管理后台（`/admin`）与普通用户端（`/portal`）独立登录入口，通过角色区分
- **双因素认证**：账号密码（第一因素）+ TOTP 动态码（第二因素）
- **手机验证码登录**：支持 Mock/生产模式灵活切换
- **角色权限体系**：基于 RBAC，角色可自定义扩展，通过角色判断管理员/普通用户身份
- **Token 机制**：统一使用 JWT（Access Token）+ Refresh Token

### 1.2 非功能需求
- 密码存储使用 BCrypt
- Refresh Token 存储在 Redis，支持主动吊销
- 登录失败次数限制，防止暴力破解
- 敏感信息加密传输（HTTPS）
- 验证码发送频率限制
- 登录日志记录（IP、设备、时间）

## 二、技术选型

| 层次 | 技术 | 选型理由 |
|------|------|----------|
| **安全框架** | Spring Security | 业界标准，生态完整，可定制性强 |
| **Token 生成与解析** | jjwt（io.jsonwebtoken） | 轻量，API 友好，支持 RSA 非对称加密 |
| **密码加密** | BCrypt（通过 Spring Security 的 PasswordEncoder） | 自带盐值，抗暴力破解 |
| **双因素认证** | TOTP 算法（aerogear-otp-java 库） | 兼容 Google Authenticator 等标准客户端 |
| **Refresh Token 存储** | Redis（Spring Data Redis + Lettuce） | 高性能，支持过期自动清理，支持分布式 |
| **验证码 Mock 切换** | Spring Profile + 策略模式 | 环境隔离，零代码改动切换 |
| **验证码存储** | Redis（带过期时间） | 自动过期，无需手动清理 |
| **登录失败计数** | Redis（AtomicLong + TTL） | 原子操作，过期自动重置 |
| **接口文档** | Apifox / Swagger | 便于前后端联调 |

## 五、核心流程设计

### 5.1 账号密码登录

```
用户 → POST /api/admin/auth/login 或 /api/portal/auth/login
      ↓
LoginRateLimitFilter（IP 级别限流）
      ↓
验证账号密码（BCrypt）
      ↓
检查登录失败次数（Redis，5次锁定30分钟）
      ↓
判断用户角色 → 校验入口合法性
  ├── 请求 /admin → 用户需拥有 SUPER_ADMIN 或 ADMIN 角色
  └── 请求 /portal → 所有角色均可
      ↓
检查是否需要2FA（用户是否绑定TOTP）
  ├── 未绑定2FA → 直接签发 Token
  └── 已绑定2FA → 返回临时 Token + 要求输入TOTP（HTTP 202）
                    ↓
              用户提交 TOTP → POST /api/auth/verify-totp
                    ↓
              验证通过 → 签发正式 Token
```

### 5.2 手机验证码登录

```
用户 → POST /api/auth/sms/send（发送验证码）
      ↓
检查发送频率（Redis，60秒内不可重复发送）
      ↓
SmsService.send(phone, code)
      ↓ 根据 Profile 自动切换
      ├── dev/test → MockServiceImpl（固定验证码 123456）
      └── prod → AliyunServiceImpl（调用阿里云 API）
      ↓
验证码存入 Redis（Key: sms:code:{phone}, TTL: 5分钟）
      ↓
用户 → POST /api/auth/sms/login（提交验证码）
      ↓
从 Redis 取出验证码比对
      ↓
签发 Token
```

### 5.3 Token 刷新流程

```
客户端 Access Token 过期（收到 401）
      ↓
POST /api/auth/refresh
Body: { refreshToken: "xxx" }
      ↓
服务端从 Redis 查询 Refresh Token
  ├── 不存在 → 需要重新登录
  ├── 存在但已过期 → 删除 Redis 记录，返回重新登录
  └── 存在且有效 → 签发新 Access Token + 新 Refresh Token（Token 轮换）
                   旧 Refresh Token 立即失效
```

## 六、核心设计决策

### 6.1 登录入口隔离方案

**不通过 `user_type` 字段，而是通过角色判断身份：**

| 角色 | 身份 | 可访问入口 |
|------|------|-----------|
| SUPER_ADMIN | 管理员 | `/admin` |
| ADMIN | 管理员 | `/admin` |
| FINANCE | 普通用户 | `/portal` |
| STAFF | 普通用户 | `/portal` |
| USER | 普通用户 | `/portal` |

**后端判断逻辑：**
```
用户角色包含 SUPER_ADMIN 或 ADMIN → 管理员 → 可访问 /admin/**
其他角色 → 普通用户 → 只能访问 /portal/**
```

### 6.2 权限控制方案

**简化但实用：前端动态路由 + 后端入口隔离**

| 层级 | 控制方式 | 说明 |
|------|----------|------|
| 页面级 | 前端动态路由 | 根据角色加载不同菜单和页面 |
| 入口级 | 后端校验角色 | `/admin/**` 需管理员角色，`/portal/**` 所有角色 |

**优点：**
- 实现简单，模块边界清晰
- 后端逻辑简洁，不需要细粒度接口权限
- 适合园区运营场景，后续可按需扩展为完整 RBAC

### 6.3 JWT 设计

**Access Token（短期，15分钟）**

| 字段 | 说明 |
|------|------|
| `sub` | 用户ID |
| `username` | 用户名 |
| `roles` | 角色编码列表（如 `["ADMIN"]`） |
| `iat` | 签发时间 |
| `exp` | 过期时间（15分钟） |
| `jti` | Token 唯一ID |

签名方式使用 **RSA 非对称加密**，私钥签发，公钥验签。

**Refresh Token（长期，7天）**
- 随机 UUID 字符串，不编码用户信息
- 存储在 Redis：`refresh_token:{userId}:{tokenValue}`
- TTL：7天

### 6.4 Token 轮换机制

每次刷新时，旧 Refresh Token 立即失效，签发新 Refresh Token。防止 Refresh Token 泄露后被长期利用。

### 6.5 短信验证码 Mock 策略

**接口抽象：**
```java
public interface SmsService {
    void send(String phone, String code);
}
```

**Profile 自动装配：**
```java
@Profile({"dev", "test"})
@Service
public class SmsServiceMock implements SmsService { ... }

@Profile("prod")
@Service
public class SmsServiceAliyun implements SmsService { ... }
```

## 七、数据库设计

### 7.1 表结构（4张表）

| 表名 | 说明 | 逻辑删除 |
|------|------|----------|
| `user` | 用户表 | ✅ |
| `role` | 角色表 | ✅ |
| `user_role` | 用户角色关联表 | 随外键处理 |
| `login_log` | 登录日志表 | 不删除，定期归档 |

**关键字段：**

- **user**：id, username, password(BCrypt), phone, totp_secret, totp_enabled, status, is_deleted
- **role**：id, role_code, role_name, description, is_deleted
- **user_role**：user_id, role_id（联合唯一索引）
- **login_log**：user_id, username, login_type, login_way, ip, user_agent, result, fail_reason

**初始化角色：** SUPER_ADMIN、ADMIN、FINANCE、STAFF、USER

### 7.2 Redis 数据结构

| Key 格式 | TTL | 说明 |
|----------|-----|------|
| `refresh_token:{userId}:{token}` | 7天 | Refresh Token |
| `sms:code:{phone}` | 5分钟 | 短信验证码 |
| `sms:limit:{phone}:minute` | 60秒 | 发送频率限制 |
| `sms:limit:{phone}:hour` | 1小时 | 每小时上限 |
| `login:fail:{username}` | 30分钟 | 登录失败次数 |

## 八、安全防护措施

| 防护点 | 措施 |
|--------|------|
| **暴力破解** | Redis 记录失败次数，连续5次锁定30分钟 |
| **验证码爆破** | 单手机号60秒内只能发送1次，每小时最多5次 |
| **Refresh Token 泄露** | Token 轮换机制，泄露后可自动失效 |
| **JWT 伪造** | RSA 非对称签名，无密钥无法伪造 |
| **密码存储** | BCrypt 加密，不可逆 |
| **密码传输** | 前端 RSA 加密 + HTTPS，双重保障 |

## 十一、面试亮点总结

1. **多模块项目结构**：父模块统一依赖管理 + 公共模块共享配置，体现工程化思维
2. **RSA 非对称签名 JWT**：区别于大多数用对称密钥的项目
3. **双 Token 机制 + Token 轮换**：体现对安全性的深入理解
4. **配置分层管理**：公共配置与环境配置分离，支持多环境切换
5. **策略模式实现 Mock/生产切换**：体现设计模式应用能力
6. **通过角色判断管理员身份**：更灵活，比固定字段扩展性更强
7. **Redis 多场景应用**：Token 存储、验证码、失败计数、频率限制
8. **与实习项目的延续性**：多级缓存、并发控制、安全加密等经验进一步深化
